# Бизнес-анализ потребностей пользователей Zabbix

**Дата:** 17 апреля 2026  
**Цель анализа:** Определить наиболее острые потребности, частые задачи и болевые точки рядовых пользователей Zabbix для последующей разработки MCP-сервера.

---

## 1. Целевая аудитория

| Роль | Типичные задачи | Уровень технической экспертизы | Частота использования Zabbix |
|------|----------------|-------------------------------|------------------------------|
| DevOps-инженер | Добавление хостов, настройка шаблонов, интеграция с CI/CD, создание триггеров для сервисов | Высокий | Ежедневно (2–4 ч/день) |
| Системный администратор | Мониторинг серверной инфраструктуры, реакция на инциденты, плановые работы, настройка агентов | Средний–высокий | Ежедневно (1–3 ч/день) |
| Разработчик | Просмотр метрик своего сервиса, отладка аномалий в продакшене, добавление кастомных метрик | Средний | 2–3 раза в неделю (30–60 мин) |
| SRE-инженер | Анализ трендов, корреляция инцидентов, постмортемы, настройка SLO/SLI через метрики | Высокий | Ежедневно (3–5 ч/день) |
| Технический директор / TL | Просмотр дашбордов доступности, подготовка отчётов, контроль SLA | Низкий–средний | Еженедельно (30–60 мин) |

## 2. Типовые сценарии использования Zabbix

| Сценарий | Частота | Длительность | Ключевые действия |
|----------|---------|--------------|-------------------|
| Просмотр дашбордов и графиков | Ежедневно | 15–30 мин | Открытие дашборда, навигация по виджетам, зум в нужный диапазон времени, переключение между хостами |
| Настройка оповещений (триггеров) | Еженедельно | 30–90 мин | Написание выражения триггера, выбор severity, настройка media-типа, тестирование (вручную) |
| Поиск и устранение инцидентов | Ежедневно | 20–120 мин | Открытие списка проблем, фильтрация, переход к графику, корреляция с другими метриками, подтверждение (acknowledge) |
| Добавление новых хостов | Еженедельно | 20–45 мин | Создание хоста, выбор групп, привязка шаблонов, настройка интерфейсов, проверка поступления данных |
| Создание кастомных метрик/шаблонов | Еженедельно–редко | 45–180 мин | Создание item с ключом, написание UserParameter или external check, создание триггера, тестирование |
| Работа с API (скрипты, интеграции) | Еженедельно | 30–120 мин | Написание JSON-RPC запросов, аутентификация, парсинг ответа, интеграция в bash/python-скрипты |
| Анализ истории метрик (корреляция, тренды) | Еженедельно | 30–60 мин | Построение графика за нужный период, выбор нескольких items, сравнение в одном графике |
| Подготовка отчётов для руководства | Редко (ежемесячно) | 60–240 мин | Экспорт данных вручную, сборка Excel/PDF, составление SLA-отчётов |
| Управление maintenance-периодами | Редко (по ситуации) | 10–20 мин | Создание maintenance, выбор хостов/групп, указание времени, подтверждение деактивации алертов |
| Массовое обновление конфигурации | Редко | 60–180 мин | Bulk-изменения через UI (неудобно), написание скриптов через API |

## 3. Болевые точки и проблемы пользователей

### 3.1. Интерфейс и UX

| Проблема | Описание | Частота проявления | Влияние |
|----------|----------|--------------------|---------|
| Сложная навигация между хостами | Чтобы найти конкретный хост, нужно знать его точное имя или группу. Поиск по частичному совпадению работает плохо в старых версиях. В крупных инсталляциях (1000+ хостов) найти нужный — задача на 3–5 минут | Ежедневно | Высокое |
| Неудобная работа с графиками | Нет возможности быстро сравнить два графика разных хостов в одном окне. Для этого нужно открывать вкладки и вручную синхронизировать временной диапазон | Ежедневно | Среднее |
| Медленный UI при большом количестве данных | При отображении графика с большим диапазоном (30+ дней) и высокой частотой сбора данных интерфейс зависает на 5–15 секунд | Ежедневно | Высокое |
| Нет глобального поиска по метрикам | Чтобы найти item с нужным ключом, нужно знать, на каком хосте он находится. Глобального поиска «найди все хосты, где есть метрика X» в UI нет | Еженедельно | Высокое |
| Запутанное меню настроек | Конфигурация разнесена по разным разделам (Configuration → Hosts, Configuration → Templates, Administration → Media types). Переключение между ними — это 5–7 кликов | Ежедневно | Среднее |

### 3.2. Настройка и конфигурация

| Проблема | Описание | Частота проявления | Влияние |
|----------|----------|--------------------|---------|
| Сложный синтаксис выражений триггеров | Для создания триггера «средняя нагрузка CPU за 5 минут > 80%» нужно написать: `avg(/hostname/system.cpu.load[percpu,avg5],5m)>0.8`. Один неправильный символ — триггер не работает без явного сообщения об ошибке | Еженедельно | Высокое |
| Невозможность протестировать триггер до его активации | Нет встроенного «preview»: нельзя проверить, сработал бы триггер за последние N часов с текущим выражением. Пользователь узнаёт об ошибке только при инциденте | Еженедельно | Высокое |
| Неочевидная логика макросов | Макросы `{$MACRO}` можно задать на уровне глобального, шаблона и хоста. Порядок переопределения не интуитивен. Пользователи часто не понимают, почему их макрос не применяется | Еженедельно | Среднее |
| Сложность добавления хоста с нуля | Для полного добавления нового хоста: создать хост → выбрать группы → настроить интерфейсы → привязать шаблоны → проверить connectivity. Это минимум 15 кликов и знание правильных шаблонов | Еженедельно | Высокое |
| Шаблоны плохо документированы | Стандартные шаблоны Zabbix не имеют встроенного описания, какие именно items и триггеры они добавляют. Пользователь вынужден открывать шаблон и изучать его вручную | Еженедельно | Среднее |
| Отсутствие dry-run при изменении шаблонов | При обновлении шаблона (например, добавлении нового item) нельзя заранее увидеть, что изменится на всех привязанных хостах | Редко | Среднее |

### 3.3. Оповещения и нотификации

| Проблема | Описание | Частота проявления | Влияние |
|----------|----------|--------------------|---------|
| Alert fatigue (усталость от алертов) | Инженер приходит утром и видит 50+ алертов. Большинство — это cascading failures (упал один сервис, посыпались зависимые). Нет автоматической группировки по корневой причине | Ежедневно | Высокое |
| Нет механизма acknowledge с комментарием через внешние системы | Подтвердить проблему (acknowledge) можно только через веб-интерфейс Zabbix. Нет удобного способа сделать это из Slack, Telegram или командной строки | Ежедневно | Высокое |
| Сложная настройка эскалации | Для настройки «если проблема не решена через 30 минут — уведомить следующего по цепочке» нужно настраивать Action с multiple operations и understanding escalation steps. Это нетривиально | Редко | Среднее |
| Нет умного подавления зависимых алертов | Если упал сетевой коммутатор, Zabbix генерирует алерты для всех хостов за ним. Механизм зависимостей существует, но требует ручной настройки для каждой пары | Ежедневно | Высокое |
| Сложность настройки maintenance (плановых работ) | Для отключения алертов на 2 часа нужно: войти в Configuration → Maintenance → Create maintenance → выбрать тип → выбрать период → выбрать хосты/группы. Минимум 8 кликов и знание нужного раздела меню | По необходимости | Среднее |
| Нет контекста в уведомлении | Стандартное уведомление по email/Telegram содержит только имя триггера и значение метрики. Нет ссылки на runbook, нет предыдущих значений, нет информации о влиянии | Ежедневно | Высокое |

### 3.4. Поиск и анализ данных

| Проблема | Описание | Частота проявления | Влияние |
|----------|----------|--------------------|---------|
| Нет возможности корреляции между несвязанными метриками | Если нужно сравнить latency сервиса с нагрузкой на БД, нужно вручную открывать два разных графика и визуально сопоставлять. Единого инструмента корреляции нет | Еженедельно | Высокое |
| Экспорт данных крайне неудобен | Для выгрузки исторических данных нужно либо строить отчёт через UI (ограниченный формат), либо писать API-запросы. Нет простого способа получить CSV за произвольный период | Еженедельно | Среднее |
| Нет аномалий-детекции из коробки | Zabbix не умеет автоматически выявлять аномалии (выбросы относительно исторической нормы). Пользователь вынужден задавать статические пороги, которые часто неактуальны | Еженедельно | Высокое |
| Трудно найти «что изменилось» перед инцидентом | При расследовании инцидента нужно вручную просматривать несколько графиков в разных разделах, чтобы понять, что изменилось за 30 минут до проблемы | При каждом инциденте | Высокое |

### 3.5. Интеграция и автоматизация

| Проблема | Описание | Частота проявления | Влияние |
|----------|----------|--------------------|---------|
| API требует много шаблонного кода | Даже простая операция «получить текущее значение метрики X для хоста Y» через Zabbix API требует: аутентификации (получить auth token), поиска item.id, запроса history.get с нужными параметрами. Это 3 HTTP-запроса и ~50 строк кода | Еженедельно | Высокое |
| Нет официального CLI-инструмента | Для автоматизации через командную строку нужно писать скрипты самостоятельно или использовать сторонние решения (zabbix-cli, zbxdb) с ограниченным покрытием API | Еженедельно | Высокое |
| Трудно интегрировать с ChatOps (Slack/Telegram) | Двусторонняя интеграция (не только уведомления, но и управление — acknowledge, создание maintenance) требует разработки собственного бота | Редко | Среднее |
| Нет нативной интеграции с Service Desk | Автоматическое создание тикетов в Jira/ServiceNow при алерте требует написания webhook-обработчика или использования сложных Action Scripts | Еженедельно | Среднее |
| Сложность Infrastructure as Code для Zabbix | Нет официальной поддержки Terraform/Ansible для управления конфигурацией Zabbix. Стороннее решение (community Terraform provider) нестабильно | Редко | Среднее |

### 3.6. Производительность и масштабирование

| Проблема | Описание | Частота проявления | Влияние |
|----------|----------|--------------------|---------|
| Медленные API-запросы при больших выборках | Запрос `history.get` за 30 дней по 10 хостам может выполняться 30–60 секунд. Нет встроенного пагинирования или стриминга данных | Еженедельно | Среднее |
| Нет кэширования данных для дашбордов | При каждом обновлении дашборда браузер делает повторные запросы к API. На больших инсталляциях это приводит к заметным тормозам | Ежедневно | Среднее |
| Трудности с bulk-операциями | Переименование группы хостов, массовое обновление macro на 50 хостах — в UI нет bulk-редактора. Нужно либо кликать по каждому хосту, либо писать скрипт | Еженедельно | Высокое |

## 4. Самые частые задачи, которые можно автоматизировать

| Задача | Текущие временные затраты | Потенциал автоматизации | Предлагаемое решение через MCP-сервер |
|--------|---------------------------|------------------------|----------------------------------------|
| Поиск активных проблем по хосту/сервису | 5–10 мин (фильтрация в UI) | Высокий | Инструмент `get_active_problems(host, severity, limit)` с естественно-языковым запросом |
| Acknowledge алерта с комментарием | 3–5 мин (открыть UI, найти проблему, нажать Ack) | Высокий | Инструмент `acknowledge_problem(problem_id, message)` |
| Создание maintenance-периода | 5–8 мин (8 кликов в UI) | Высокий | Инструмент `create_maintenance(hosts, duration_minutes, reason)` |
| Добавление нового хоста с шаблоном | 15–20 мин | Высокий | Инструмент `add_host(name, ip, templates, groups)` — один вызов вместо 15 кликов |
| Получение текущего значения метрики | 3–7 мин (3 API-запроса или навигация в UI) | Высокий | Инструмент `get_metric_value(host, item_key)` |
| Построение отчёта доступности за период | 60–240 мин (ручной сбор данных) | Высокий | Инструмент `get_availability_report(hosts, period)` с готовым форматированием |
| Поиск хостов по шаблону/группе/тегу | 5–15 мин (навигация в UI) | Высокий | Инструмент `search_hosts(query, group, template, tag)` |
| Получение истории метрики за период | 10–30 мин (API-скрипт) | Средний | Инструмент `get_metric_history(host, item_key, from, to)` с возвратом в удобном формате |
| Создание триггера с проверкой синтаксиса | 30–90 мин (итеративное тестирование) | Средний | Инструмент `create_trigger(description, expression, severity)` с валидацией выражения и preview |
| Массовое обновление макросов на хостах | 30–60 мин (ручное) | Высокий | Инструмент `bulk_update_macro(hosts_pattern, macro_name, value)` |
| Проверка состояния агента на хосте | 5–10 мин (переход в раздел Hosts → проверка availability) | Высокий | Инструмент `check_host_availability(host)` |
| Экспорт метрик в CSV/JSON | 30–60 мин (API-скрипт) | Средний | Инструмент `export_metrics(hosts, items, period, format)` |

## 5. Возможности для улучшения пользовательского опыта

| Область улучшения | Текущая проблема | Предлагаемое решение | Приоритет |
|-------------------|------------------|----------------------|-----------|
| Управление алертами через чат | Нет возможности acknowledge и создать maintenance без открытия UI | MCP-инструменты для ChatOps: acknowledge, snooze, maintenance через LLM-интерфейс | High |
| Естественно-языковые запросы | Пользователь должен знать синтаксис API и ключи метрик | LLM маппирует «покажи нагрузку CPU на сервере prod-web-01 за вчера» в API-вызовы | High |
| Быстрая диагностика инцидентов | Инженер тратит 20+ мин на сбор контекста при инциденте | Инструмент «incident summary»: сколько проблем, какие хосты, какие триггеры, временная линия | High |
| Упрощение onboarding нового хоста | 15+ кликов и знание структуры шаблонов | Один вызов с авто-подбором шаблонов на основе роли хоста (web, db, cache) | High |
| Упрощение создания триггеров | Сложный синтаксис, нет тестирования | LLM генерирует выражение по описанию + проверяет его на исторических данных | Medium |
| Сводный отчёт для руководства | Ручной сбор и форматирование, 2–4 часа | Автоматическая генерация markdown/PDF отчёта с SLA, топ инцидентов, трендами | Medium |
| Поиск метрик по смыслу | Нужно знать точный ключ item (например, `net.if.in[eth0]`) | Поиск по описанию: «входящий трафик на сетевом интерфейсе» → найти нужный item | Medium |
| Умный фильтр алертов | 50 алертов утром, большинство — cascading failures | Группировка проблем по предполагаемой корневой причине через анализ зависимостей | High |

## 6. Требования к MCP-серверу Zabbix

### 6.1. Функциональные требования

| ID | Требование | Приоритет (MoSCoW) | Обоснование |
|----|------------|-------------------|-------------|
| F-01 | Получение списка активных проблем с фильтрацией по хосту, группе, severity | Must | Ежедневная задача каждого инженера |
| F-02 | Подтверждение (acknowledge) проблемы с комментарием | Must | Устраняет необходимость открывать UI для ключевого действия при инцидентах |
| F-03 | Создание maintenance-периода для хостов/групп с указанием продолжительности | Must | Частая задача при плановых работах; текущий UI требует 8+ кликов |
| F-04 | Добавление нового хоста с привязкой шаблонов и групп | Must | Ключевая задача DevOps; автоматизация сокращает время с 20 до 1 мин |
| F-05 | Получение текущего значения item по ключу для указанного хоста | Must | Базовая операция диагностики; требует 3 API-запроса без MCP |
| F-06 | Получение истории метрики за произвольный период | Must | Необходима для анализа инцидентов и построения отчётов |
| F-07 | Поиск хостов по имени, группе, шаблону, тегу | Must | Навигация в крупных инсталляциях занимает значительное время |
| F-08 | Получение списка triggers с их статусом для хоста | Should | Помогает диагностировать, какие условия мониторинга настроены |
| F-09 | Создание/обновление trigger с валидацией синтаксиса выражения | Should | Снижает ошибки при настройке мониторинга |
| F-10 | Получение сводки по инциденту (affected hosts, timeline, triggered items) | Should | Критично для быстрой диагностики; экономит 15–30 мин при каждом инциденте |
| F-11 | Bulk-обновление macros для группы хостов | Should | Частая задача при изменении конфигурации окружения |
| F-12 | Генерация отчёта доступности (availability report) за период | Should | Замена ручного сбора данных для руководства |
| F-13 | Поиск items (метрик) по имени/ключу/описанию через хосты | Could | Упрощает работу с незнакомыми шаблонами |
| F-14 | Проверка доступности (availability) агента Zabbix на хосте | Could | Быстрая диагностика проблем с агентом |
| F-15 | Экспорт метрик в структурированный формат (JSON/CSV) | Could | Нужен для интеграции с внешними системами |

### 6.2. Нефункциональные требования

| ID | Требование | Описание |
|----|------------|----------|
| NF-01 | Безопасность аутентификации | MCP-сервер должен поддерживать хранение Zabbix API-токена в переменных окружения, без хранения в открытом виде в конфигах |
| NF-02 | Производительность | Инструменты должны возвращать результат за ≤5 секунд для стандартных запросов (≤100 объектов). Поддержка пагинации для больших выборок |
| NF-03 | Совместимость | Поддержка Zabbix API версий 6.0 LTS и 6.4, 7.0 (наиболее распространённые в продакшене) |
| NF-04 | Обработка ошибок | При ошибке API возвращать понятное сообщение на человеческом языке, а не raw JSON error от Zabbix |
| NF-05 | Идемпотентность | Операции создания (хост, maintenance) должны проверять существование объекта перед созданием и возвращать существующий ID вместо ошибки |
| NF-06 | Логирование | Все вызовы инструментов должны логироваться с timestamp, параметрами и результатом для аудита |
| NF-07 | Конфигурируемость | URL Zabbix API, таймауты, лимиты пагинации — всё должно быть конфигурируемым через переменные окружения |

### 6.3. Инструменты и возможности MCP-сервера (ранжированные по важности)

1. **`get_active_problems`** — получение активных проблем с фильтрацией: самый частый use case каждого инженера
2. **`acknowledge_problem`** — подтверждение алерта с комментарием: устраняет необходимость открывать UI при инцидентах
3. **`create_maintenance`** — создание maintenance-периода: часто нужный, но неудобный в UI сценарий
4. **`add_host`** — добавление хоста с шаблонами: ключевая операция DevOps-пайплайна
5. **`get_metric_value`** — текущее значение метрики: диагностика в реальном времени
6. **`get_metric_history`** — история метрики за период: анализ инцидентов и трендов
7. **`search_hosts`** — поиск хостов по параметрам: навигация в больших инсталляциях
8. **`get_incident_summary`** — сводка по инциденту: критично при response на алерты
9. **`create_trigger`** — создание триггера с валидацией: снижает ошибки в конфигурации
10. **`get_availability_report`** — отчёт доступности: замена ручной работы для отчётности
11. **`bulk_update_macro`** — массовое обновление макросов: автоматизация рутинных изменений
12. **`search_items`** — поиск метрик по описанию: устраняет зависимость от знания точных ключей

## 8. Выводы и рекомендации для разработки MCP-сервера

### Топ-3 критических потребности:

1. **Управление инцидентами без UI** — возможность получить список активных проблем, acknowledge алерт и создать maintenance через LLM-интерфейс. Это устраняет самый частый фрикцион: необходимость открывать браузер и кликать по UI при каждом инциденте, что особенно болезненно в ночное дежурство.

2. **Быстрое добавление хостов и получение метрик** — onboarding нового сервиса в мониторинг и диагностика «что сейчас происходит с метриками» — это задачи, которые выполняются ежедневно и требуют либо знания UI, либо написания API-кода. MCP-инструменты делают это доступным без специализированных знаний.

3. **Контекстная диагностика инцидентов** — инструмент `get_incident_summary`, который за один вызов возвращает: когда началось, какие хосты затронуты, какие триггеры сработали, динамику метрик. Сейчас это занимает 20–30 минут ручного анализа.

### Рекомендуемый MVP MCP-сервера:

- **5 инструментов первой очереди:** `get_active_problems`, `acknowledge_problem`, `create_maintenance`, `add_host`, `get_metric_value`
- Поддержка Zabbix API 6.0 LTS и 7.0
- Конфигурация через переменные окружения (`ZABBIX_URL`, `ZABBIX_API_TOKEN`)
- Понятные error messages на случай недоступности API или неверных параметров
- Базовое логирование вызовов

### Долгосрочные возможности для развития:

- **Естественно-языковое создание триггеров**: LLM генерирует валидное Zabbix expression на основе описания на русском/английском с проверкой на исторических данных
- **Умная группировка алертов по корневой причине**: анализ зависимостей хостов и автоматическое определение root cause среди каскадных сбоев
- **Интеграция с ChatOps**: готовый Telegram/Slack bot поверх MCP-инструментов для полного управления Zabbix через мессенджер
- **Anomaly detection**: инструмент, который анализирует историческое поведение метрики и предупреждает о нетипичных значениях без настройки статических порогов
- **Infrastructure as Code bridge**: синхронизация конфигурации Zabbix с Git-репозиторием через MCP (версионирование хостов, шаблонов, триггеров)
- **Автоматическая генерация отчётов**: markdown/PDF отчёт с SLA, топ-N инцидентами, трендами доступности — по расписанию или по запросу
